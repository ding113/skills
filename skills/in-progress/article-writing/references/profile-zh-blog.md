# 中文技术博客风格画像（个人博客／团队分享／综述）

**速览**：动笔前先选人称立场——我（亲历反思）／我们（团队复盘）／第三人称（中立综述），滑动须有动机。
开头用论断、自我暴露或场景悬念破题；正文按「问题→方案→撞墙→修补」推进；收尾回到人的处境，禁「总而言之」。
比喻必须推进论点，数字必带换算与来源，负向数据照登；主引号一律直角引号「」，嵌套用『』。

## 一、人称模式：先定叙事立场

| 模式 | 适用场景 | 语气特征 |
| --- | --- | --- |
| A「我」 | 亲历认知转变、学习笔记、个人观点文 | 坦诚、自我暴露、带反思；敢写「我错了」「我以为我懂了」；技术判断包裹在个人经历里 |
| B「我们」 | 团队实践复盘、系统落地分享 | 集体经验复盘，谦逊但有底气；「抛砖引玉」姿态＋大量工程细节与数据背书；比 A 少情绪、多结构 |
| C 第三人称 | 概念综述、编译导读、行业研报 | 中立、克制、信息密度优先；作者隐身，自称只用「本文」「笔者」；让事实与来源说话 |

人称允许滑动，但滑动必须有动机：讲亲身经历用「我」，讲团队产出用「我们」，讲客观机制去人称。真实语料里纯单一人称的文章反而少，常见路径是个人开场破冰→「我们」讲实践→无人称讲原理。

各模式的纪律差异：
- A 类：口语词有限度使用（「说来惭愧」「吃灰」「扒了一遍」），语气词（啊／呢／吧）几乎不用；短句收段但每句成分完整——口语感靠节奏，不靠省略主语宾语；技术表述用因果关联词织进叙事（「写到一半发现行不通，因为……这导致……」），不做「口语一句＋术语一句」的硬拼；自谦必须落在具体事实上（「IDEA 吃灰半年」），空泛自谦无效，且同一个自嘲梗全文只用一次——开头用过的梗不拿到结尾再抖一次。
- B 类：避免第一人称情绪词、反问连用、语气词；高频动词群为落地、沉淀、收敛、拉齐、兜底、演进（「赋能」慎用）；谦辞是固定件（「抛砖引玉」「希望对正在做类似事情的同学有用」）。
- C 类：称呼语、感叹号、情绪词清零；放弃比喻和拟人，修辞只剩设问与排比；数据必带来源，断言必带范围限定词；人物首次出现必带完整头衔定语（「LWN.net 知名作者、Linux 内核文档核心维护者 Jonathan Corbet」）。

## 二、标题设计模式（均为真实标题）

- 论断先行（冒号后是一个判断）：《Qoder 工程实践：当瓶颈从模型转移到人》。
- 情绪化推荐式（导读获客位）：《这大概是我读过最硬核的一次 Linux 内核重构文章了》——标题与正文人称允许分离，「我」式标题拉流量，正文可以是中立编译体。
- 主体＋如何（团队文占比最高）：《B站是如何实现原声视频翻译的》。
- 从 A 到 B（用迁移路径命名）：《从AI Coding到Harness Engineering的端到端工程开发实践》。
- 观点句＋破折号＋文体说明：《Harness不是目的，知识才是护城河 —— 一个AI工程交付团队的知识沉淀实践》。
- 规模词＋领域（自我标注体量）：《万字长文谈B站变更管控体系设计与实践》；【揭秘】式标题党语料中仅 1 例，视为可选而非默认。
- 主标题定域、副标题提问：《AI Native 时代 —— 研发组织何去何从》。
- 综述型三节奏名词短语（标题即目录）：《Loop Engineering 概念解析、思考与实践》。

标题硬约束（违反任意一条即重拟）：核心技术词必须出现（写 Redis 缓存的文章标题里要有「缓存」）；不堆裸指标与行话（「P99 从 480 毫秒到 96 毫秒」对外行毫无吸引力，数据进标题须换算成外行能感到分量的形态）；标题许诺读者拿得走的东西（排查过程、方法、教训），不是作者的轶事动机（「被老板艾特的一周」不是卖点）；总长度控制在一眼读完的范围，超过 25 字就该做主副标题拆分。

子标题体系：A 类用编号大写（`01`/`02` 独立成行）或动宾对仗短语（《从「更快打字」到「任务自主交付」》），散文式科普可用悬念半句；B 类用「一、二、三」＋名词短语，二级用 1.1/1.2，或章回式「第一章：……」。

## 三、结构与流程惯例

- A 类开场三件套：一个加粗论断＋一段亲历/具体数字＋一个反直觉的个人反应。B 类开场 ≤3 段讲清：我们是谁→做了什么→为什么值得你读。C 类开场＝领域背景一段＋主角定位一句＋权威背书。
- 首段或首节末给全文导航；超过一万字必须给跳读建议，并说明哪几章密度最高——等于替读者做一次内容评级。
- 痛点先行：困境列成 3 点左右，每点「**加粗短语：**现象→原因→代价」，同构并列。
- 正文推进＝问题→方案→撞墙→修补，坦率写失败设计；负向指标与正向指标同等待遇加粗列出——全程成功叙事是汇报 PPT，不是技术文章。
- 每个架构给图，图前一句引导（「整体的框架逐渐演变成下图所示：」），不让图裸奔；横向对比给三列表（对象／策略／一句话概括——第三列是提炼列）。
- 金句加粗，一篇 3~8 处（约每 1.5~2 屏一处），加粗的是判断而非名词。
- 锚点回收：开头埋的比喻/问题/数字，中后段用「回到……」显式召回（「回到最开始。」「回到那个"实习生"的比喻」），制造结构闭环。
- 章节过渡自带信息量：「到这里，问题定义清楚了。」收口上一章；「有了……，……就……了」把上一节产出变成下一节前提。禁止空转过渡。
- 收尾：A 类回到人的处境或反问，禁「总而言之／综上所述」；B 类三件套＝展望一段＋参考/往期＋「欢迎留言一起参与讨论~」（互动语另起一行，不与正文论证混排）；C 类文末参考文献列表，不做互动收尾。

## 四、开头破题手法

1. **论断先行，认知转变作钩子**——论断要敢下且全文能兑现，加粗后立刻拉回个人：
   > 当 AI 输出的价值稳定超过 Token 成本之后，**瓶颈从模型能力转移到了人的精力**。……这个认知改变了我过去半年的工作方式，这种改变的发生不是渐变式的，是某天突然看清楚了。（Qoder 文）
2. **学习动机开场＋差异化声明**——「我以为我懂了，直到」制造认知落差，再一句讲清本文与别人的不同：
   > 看了很多的文章和视频，我以为我理解大模型的工作原理了，直到看了vLLM的代码，我发现很多地方理解的太过表面。因此花了大概2个月的业余时间，深入阅读了vLLM的源码，本文算是对于学习代码的一个总结。（AI Infra 文）
3. **自我暴露式破冰**——自嘲落在具体事实上，段尾接一个时代级短判断完成升维：
   > 说来惭愧，作为一个校招入职不到两年的程序员，我的 IDEA 已经吃灰半年多了。那些曾经引以为傲的快捷键肌肉记忆，如今被同事们调侃为"古法编程"。大模型这波浪潮，不是未来已来，是过去已去。（支付宝618 文）
4. **团队自报家门**——两句内讲清「我们是谁＋发布了什么」，随后立刻转入「这意味着什么」：
   > 大家好，我们是 BILIBILI Index 团队。……最近，我们上线了一个新能力：支持将部分中文视频翻译为外语的原声风格配音。（B站翻译文）
5. **宏观背景一段压到痛点清单**——背景只给一段，段尾必须压到具体痛点或数据，不许背景漫游：
   > 过去几年，抖音的业务边界持续拓宽，从核心短视频延伸至直播、电商、本地生活等多元场景，用户规模与交互复杂度持续增长。设备端需同时承载高性能渲染、低延迟推流、多线程预加载等密集任务，内存峰值与并发任务数激增，传统资源调度机制难以为继，面临以下挑战：（抖音文）
6. **场景悬念开场**——先给画面再揭机制，悬念不能拖，下一段就要点破：
   > 凌晨三点，没有人在群里催进度，也没有人在电脑前等结果。（AI军团文）

势能构建与挖坑（配合上面六式使用）：
- 结论先行摆成果，给读者读下去的理由：「先看结果——下面是我们在这场大促中交出的答卷：」（支付宝618 文）。
- 先亮结论，再声明「走一遍到达它的路」，合法化后文按认知阶段推进（见 P9）。
- 递进设问引导读者自己推导出方案，再揭晓名字（见 P5），把「介绍」变成「发明」。
- 「问题来了」式转轴，用真问题换挡：「那问题来了，既然Agent Loop 早就有了，为什么最近各路 AI 大佬又开始密集讨论它？」（Loop 文）。
- 「**为什么……？**」加粗小标题自问自答，把读者最可能的质疑先写成标题：「**为什么阈值是 60 / 80 / 95？**」「**为什么先 Snip 再 Prune？**」（压缩策略文）。
- 长文导航给跳读路径：
  > **全文阅读约 25 分钟。**一共 10 小节，按"先讲项目 → 再讲方法论 → 再讲撞墙 → 最后讲取舍"的顺序展开。如果时间紧，可以直接跳到第六章（门禁脚本）和第七章（Team Mode 撞墙复盘）——这两章的密度最高、踩坑最深，对其他团队可能也最有参考价值。（大仓文）

## 五、段落递进模板 P1–P11

- **P1 论断加粗→个人认知变化→具体数字（带换算）→反直觉个人反应**：个人观点文开头的四步，论断给方向、认知变化给动机、数字给锚点、反应给人味。
  > Peter（OpenClaw作者） 一天内提交了 627 次代码——计算一下，一天有 1440 分钟，这意味着每次代码提交间隔平均不到 2.3 分钟。我看到这个数字的第一反应不是佩服，是替他感到累，这一天结束后，他还剩多少判断力？（Qoder 文）
- **P2 场景悬念→变化清单→揭示机制→上升到论点**：先泄底防读者猜谜，再用「真正的变化不在于 A，而在于 B」兑换成论点。
  > 凌晨三点，没有人在群里催进度，也没有人在电脑前等结果。……这些事情当然是 Agent 在做。（AI军团文）
- **P3 抽象定义→排比实例链→迁移到本文主题**：先掐掉玄学感，用三个同构实例铺排，最后一句迁移到本文对象。
  > 水分子本身没有"波浪"，神经元本身没有"意识"，单只蚂蚁没有"城市规划"，但当它们以某种密度、规则和反馈连接起来，整体就会出现新的层级。（涌现文）
- **P4 反常数据清单→节奏点评→追到根本问题**：数据要有反差，点评跳出数字升一层，再把话题拉到第一性问题。
  > 更值得停下的不是数字，是节奏：一个工程师上午 10 点上线一个新功能、中午做 A/B 测试、下午 3 点根据数据下线、5 点上线更好的版本。同一天。这是过去 6 周才能完成的迭代。（AI Native 文）
- **P5 铺陈机制→逼出核心矛盾→设问→「恭喜你发明了 X」**：把既有方案还原成读者自己能推出来的必然解，前提是推导链真的自然。
  > 有没有可能将调度的从request level下沉到token level呢？ 恭喜你发明了continuous batching。（AI Infra 文）
- **P6 存量盘点→比喻→能力边界→现实梗阻**：比喻只负责翻译处境，「更现实的是」再压一层约束，痛点递进两级。
  > 它们的单点能力都已经不弱——像是一群刚入职的优秀实习生：让他们做手上这一摊事，做得不错；但你不能指望他们自己知道这件事什么时候该开始、做完之后该交给谁、卡住了该找谁兜底。（AI军团文）
- **P7 加粗痛点导语→现象→机制后果**：痛点并列段的标准细胞，三五个同构并列，每个不超过一段。
  > **声音人格的缺失：**传统配音虽解决语言障碍，却抹去了创作者独特的音色、语调与口音——这些正是"谁在说话"的核心标识。在人格化传播时代，声音是IP的重要组成部分。一旦被标准化配音取代，情感连接断裂，影响力随之衰减。（B站翻译文）
- **P8 决策自问自答：问句加粗→逐个数字给理由→给退路**：每个魔法数字给两侧边界理由，最后交代逃生通道（可配置/可禁用）。
  > 60% 是预防性维护的甜点——再低会频繁触发没意义，再高模型注意力已经开始下降。80% 是危险线，离溢出还有缓冲但已经不能等。95% 是最后防线，再不调 LLM 就要爆了。（压缩策略文）
- **P9 先亮结论→声明「走一遍到达它的路」→认知阶梯**：结论提前锁定预期，为全文按阶段推进授权，并预告每段都有弯路可看。
  > 每一次工作方式的进化，都在回答同一件事：**怎样用更少的人力在线时间，让更多的 Token 持续流动？**我认为这个问题的答案不是一个更好的 IDE 插件，也不是一个更聪明的模型，是一整套云端持续进化的 Harness 基础设施。但在讲那个思考结论之前，我想先走一遍到达它的路。（Qoder 文）
- **P10 撞墙复盘：预告价值→症状→排查**：先声明故事的教训级别（不只是修 bug，是一类陷阱），再按事故报告结构走，症状精确到用户动作。
  > 下面这个故事是我们 Harness 演进里最关键的一次"撞墙"，值得单独展开讲。它的价值不只在于修了一个 bug，更在于让我们彻底理解了一种很容易踩进去的陷阱——"复杂度自给自足"。（大仓文）
- **P11 反面教材先行：旧做法一句话→痛点根源清单→总结病根**：旧逻辑压缩成一句话，痛点列成有限清单并预告「新方案就是在解这几条」，末尾收敛成一个病根句。
  > 第一代做法的根本问题：它把压缩当突发事件处理，而不是一种持续维护的能力。（压缩策略文）

## 六、术语与标注处理惯例

- 中英文之间、数字与汉字之间加半角空格是主流；按发布平台惯例可配置（B站系不加）。
- 专有名词首次出现给最小限度身份括注：「Peter（OpenClaw作者）」；圈内名人直接引不注释，默认读者圈层知道。
- 英文概念首次出现给中文对照；缩写用「（以下简称 X）」一次性声明：「Kubernetes（以下简称 K8s）」。
- 内部系统首次出现给全称＋紧跟一句定位，先用再解释：「Douyin Dynamic Performance Framework（DDPF 应运而生）」。
- 外部术语用隐喻式释义而非词典式：词源＋「就像……」一句话同时讲清「是什么」和「不是什么」（如 harness＝马具→引导和约束而非增强）。
- 注释主流是文内括注，用于身份说明、即时换算（「计算一下，一天有 1440 分钟」）、限定声明、括注即时举例（「（想买个蓝牙耳机，200 以内/有没有手机券）」）。不用 Markdown 脚注 `[^1]`；需要引用时用手写角标 \[1\]＋文末参考条目（作者、标题、出处、日期＋链接）。
- 引用外部分享/演讲：正文内「注：」起头＋紧跟裸链接；引用他人观点整句用斜体＋引号。
- 数据必带来源：「据谷歌SRE统计，高达70%的生产事故与线上服务的变更直接相关。」数字不自证，必须换算成读者可感的粒度；改动价值用前后两个具体数字说话，紧跟改动描述，不攒到文末。

## 七、反面特征清单（审校时逐条对照，命中即改）

- [ ] 「随着人工智能的飞速发展」式宏大空话开场。
- [ ] 「总而言之」「综上所述」式收束。
- [ ] 无信息量的过渡句：「接下来让我们一起看看」。
- [ ] 每段等长、三段式排比成瘾。
- [ ] 全文无一处具体数字、无一个具名的人/系统/时间（真文章每篇都有大量具体锚点：627 次提交、5%~20% 提升、Kairui Song）。
- [ ] 「我」类没有个人经历，只有观点罗列（洗稿感）。
- [ ] 「我们」类全程成功叙事、没有失败复盘、负向指标被藏起来（汇报 PPT 感）。
- [ ] 「三」类出现「我觉得」「大家好」或感叹号（立场泄漏）。
- [ ] 比喻只做装饰不推进论点，或在比喻里逗留不回机制。
- [ ] 魔法数字（阈值/参数）不解释为什么是这个数、不给两侧边界。
- [ ] 空泛自谦（「水平有限」）而非具体事实自嘲。
- [ ] 引完权威就走，不接自有推演。
- [ ] 「不是 A，而是 B」句式出现在开头段，或全文超过一处（该句式是破与立最套路化的表面形态，逻辑对也不行）。
- [ ] 「不是 A 而是 B」的 A 是没人真那么想的稻草人。
- [ ] 句子成分残缺：为追求短句凌厉而省掉主语/宾语，读者需要自行补全「谁、对什么、结果如何」。
- [ ] 「口语一句＋术语一句」硬拼，无因果关联词缝合，念出来不像人说话。
- [ ] 通篇句号匀速推进，中间突兀插入反问/感叹，情绪起伏像节拍器而非叙事。
- [ ] 结尾抖机灵回调开头的梗，或用「希望大家」式号召收束。
- [ ] 图表裸奔：图前没有引导句说明图的身份。
- [ ] 语气词（啊/呢/吧）、感叹号、网络梗密集出现（真语料克制到几乎为零）。

## 八、标点提醒

- 主引号一律直角引号「」，嵌套用『』；「」标术语与概念（「有品味」「更快打字」），与引用他人原话的引号分工。
- 全角标点为基底；数字、百分比、单位一律半角（「30%」「2.3 分钟」）。
- 破折号「——」高频用于解释补充与转折递进；省略号极少用。
- 顿号管并列，分号管长并列句，冒号引导定义与清单；「**加粗短语：**正文」是行内小标题的标准形态。
